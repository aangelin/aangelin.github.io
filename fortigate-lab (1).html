// overview

This lab covers the full deployment lifecycle of a FortiGate-VM — from sourcing the correct image and navigating hypervisor compatibility issues, through CLI-based interface configuration, GUI access, and FortiCloud license activation, to building and ordering a real firewall policy set. The goal was to get hands-on with FortiOS fundamentals: CLI syntax, zone-based policy logic, Internet Service objects, and traffic logging.

// deployment steps

  1. Downloaded FortiGate-VM OVF package (v7.4.12) from Fortinet's VM image download portal using a free FortiCare account
  2. Imported OVF into VMware Workstation Pro — configured port1 as Bridged/NAT (WAN) and port2 as Host-only VMnet1 (LAN)
  3. Completed first-boot login via VMware console, set admin password, and configured port2 with a static IP for management access
  4. Accessed the FortiGate web GUI at https://192.168.122.254 from the host browser
  5. Activated the permanent evaluation license via FortiCloud credentials in the GUI — FortiGate rebooted and came up fully licensed
  6. Verified internet connectivity from FortiGate CLI with execute ping 8.8.8.8 — 0% packet loss confirmed

// firewall policy configuration

Built a structured three-tier policy set demonstrating proper rule hierarchy — specific deny rules above general allow rules, with FortiGate's implicit deny-all as the final catch. Used FortiGate's Internet Service Database (ISDB) to block Facebook by service category rather than raw IP, which is a more realistic and maintainable enterprise approach.

# Policy Name Source Destination Action Log
1 Block-Facebook all Facebook-Web, DNS, ICMP ⊘ DENY ✓
2 LAN-to-WAN-Allow all all ✓ ACCEPT ✓
— implicit_deny all all ⊘ DENY (default) —

Rule ordering is critical: FortiGate evaluates policies top-to-bottom and matches on the first hit. The Block-Facebook deny rule must sit above the general LAN-to-WAN allow rule, otherwise Facebook traffic would be permitted before the deny is evaluated.
